deleted (so deal-signal-mcp tools are NOT currently reachable from the
agent). To recreate for Hermes:

- Hermes registers MCP servers via `mcp_servers:` block in
  `/opt/data/config.yaml` (NOT via a sibling ConfigMap the way OpenClaw
  did). Two paths:
  1. **Bake into the bootstrap-config init container** — add a third
     value source alongside `MODEL_DEFAULT` / `MODEL_PROVIDER` /
     `MODEL_BASE_URL` for the deal-signal MCP URL, and have the init
     script overlay it into `model.mcp_servers` (or wherever Hermes
     reads MCP config in cli-config.yaml).
  2. **Run `hermes mcp add`** (if it exists; check
     `/opt/hermes/.venv/bin/hermes mcp --help`) at runtime, with the
     deal-signal-mcp service URL.

  `http://deal-signal-mcp.deal-signal.svc.cluster.local:8080/mcp/`
  (transport: streamable-http) — same URL that was in the deleted
  openclaw-mcp-extra ConfigMap. That endpoint is unchanged.

- Per the earlier research, Hermes documentation was inconsistent on
  whether HTTP-transport MCP works end-to-end, so verify with a tool
  call after you wire it up. If HTTP doesn't work, fall back to a

### 2. Consider the openclaw namespace rename

The `openclaw` namespace is **still there**, holding the RAG pipeline
that everything depends on:

```
NAME                                 STATUS    AGE
chromadb-5d9db9bd4f-h8jdb            Running   10d
rag-indexer-5567dcd468-vg97k         Running   36h
rag-mcp-6b9dfd889d-ws56j             Running   36h
```

Plus PVCs: `chromadb-data`, `obsidian-vault` (still mounted by
rag-indexer), `obsidian-sync-config` (orphan, harmless).

The namespace name is the only remaining "sign of OpenClaw." Renaming
options, in order of disruption:

**A. Rename via Argo Application destination edit + manual PVC migration.**
   Edit `clusters/default/applications/children/{chromadb,rag-indexer,rag-mcp,obsidian-sync}.yaml`
   to set `destination.namespace: rag` (or whatever name). Argo creates
   the new namespace, deploys there, prunes old. ChromaDB data is in a
   local-path PVC — recreating in the new namespace gives you a fresh
   empty PVC. **Re-index cost.**

**B. Migrate PVC data manually before namespace flip.**
   For chromadb data preservation: scale chromadb to 0, ssh to storage
   node, `cp -a /var/lib/rancher/k3s/storage/<old-pvc>/data /var/lib/rancher/k3s/storage/<new-pvc>/data`,
   apply new manifest, scale up. Tedious but no re-index.

**C. Leave it alone.**
   Functional cost: zero. Cosmetic cost: `kubectl get ns` shows
   "openclaw" forever. Easiest.

I'd suggest doing this in tandem with #1 — once you have hermes-vault
working everywhere, repoint rag-indexer at it, drop the legacy
`obsidian-sync` Argo Application, and rename the namespace.
